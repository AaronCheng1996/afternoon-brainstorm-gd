## 2026-07-18（使用者實機部署 · **修正 2 個實跑才會炸的 bug**）

   重新載入較嚴格而炸出。**修正**：`for layer: Node2D in […]`／`var tw: Tween =`。

2. **`@rpc` 節點路徑兩端不一致，握手永遠 "Node not found"**：`server_main.gd` 把 MultiplayerAPI
   根設為伺服器 peer 節點自己的路徑（令其相對路徑＝空字串），`online_lobby.gd` 卻把 `NetClient`
   掛在場景節點下（相對路徑＝兩段 `"OnlineLobby/NetPeer"`）——兩端算出的 RPC 定址路徑對不上。
   **修正**：兩端統一採用 `net_peer_base.gd` 原本就寫明的設計——`server_main` 根改 `^"/root"`、
   `online_lobby` 的 `NetClient` 改掛 `get_tree().root`（並補 `_exit_tree()` 安全網防止不再隨場景
   自動釋放而外洩）。**用真實兩程序 ENet smoke test 驗證**（非既有的同程序 loopback-bus 測試——
   那批測試繞過真實 RPC 定址，正是這個 bug 沒被既有 2426 個測試抓到的原因）：headless
   `server_main.gd` 開埠，另一程序的真實 `NetClient` 連線送 hello，成功收到 `welcome`。

兩項修正後：**2426 passed／洩漏 39/78 零新增**（既有套件無回歸）。

**留意**：既有 net 測試手法（覆寫 `_transmit` 直接同程序投遞）對「兩端 RPC 節點路徑是否對得上」
是**盲區**——這類問題只有真實 `@rpc`-over-ENet 才會暴露。若之後再改 `server_main`/`online_lobby`/
`net_peer_base` 的節點掛載方式，建議比照本次的真實兩程序 smoke test 手動驗證一次
