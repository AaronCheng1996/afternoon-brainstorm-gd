# 04 Godot 目標架構設計

> 對應決策 D1–D6。本文是「怎麼蓋」的規格；「照什麼規則蓋」在 01/02/03。

## 1. 分層總覽

```
┌─ presentation（Node 世界，可換皮）────────────────────┐
│  scenes/battle/  棋盤、棋子視圖、手牌、HUD、動畫排程器   │
│  scenes/menu/ scenes/draft/ scenes/end_game/ ...        │
└──────────────▲──────────────────────────────────────────┘
               │ GameEvent 佇列（單向）        ▲ GameAction（單向）
┌─ core（純 GDScript RefCounted，零 Node 依賴）───────────┐
│  game_core.gd      狀態 + 回合引擎 + 行動分派            │
│  combat.gd         傷害管線 + 攻擊佇列                   │
│  ability_system/   能力、觸發、狀態(Status)              │
│  faction/          Token/Luck/Totem/Coin/Shadow 引擎     │
└──────────────▲──────────────────────────────────────────┘
               │ 讀取
┌─ data ──────┴───────────────────────────────────────────┐
│  data/balance/*.json（由 Python repo 同步，見 05）        │
│  data/abilities/*.gd 或 .tres（能力定義，程式碼即資料）    │
│  BalanceDB autoload：載入、驗證、查詢、版本 hash          │
└──────────────────────────────────────────────────────────┘
ai/       ai_query.gd ai_evaluator.gd strategies/ ai_controller.gd（只依賴 core）
modes/    campaign/ endless/（流程編排與存檔）
tests/    run_tests.gd + test_*.gd（headless）
```

**鐵律**：core 與 ai 裡不得 `extends Node`、不得 `get_tree()`、不得 `load("res://scenes/...")`。
唯一與外界的介面：吃 `GameAction`，吐 `GameEvent` 陣列 + 可查詢狀態。

## 3. GameCore 介面草圖

```gdscript
class_name GameCore extends RefCounted

var config: GameConfig            # 棋盤大小、win_threshold、常數
var board: BoardState             # occupy 格狀態
var players: Array[PlayerState]   # [p1, p2]
var neutral_pieces: Array[PieceState]
var score: int
var turn_number: int
var rng: RngService               # 有 seed，全部隨機經由它
var stats: Statistics
var event_sink: Array[GameEvent]  # 本步驟產生的表現層事件
# 各 faction 引擎與 §「01 §2」全部欄位…

func setup(p1_deck: Array[String], p2_deck: Array[String], seed: int) -> void
func dispatch(action: GameAction) -> ActionResult   # 01 §4 的全部行動
func logic_step() -> void                           # 01 §3 的 logic_update（回收/補抽）
func drain_events() -> Array[GameEvent]             # 表現層每幀取走
func is_over() -> bool;  func winner() -> int
func clone() -> GameCore                            # Phase 5（claude AI）才需要
```

`PieceState` 為純資料（RefCounted）：`card_id, owner, pos, health, max_health, damage,
original_damage, armor, extra_damage, attack_uses, attack_types, statuses:{}, instance_id,
hit_cards:[], pending_death, shadows:[]（Fuchsia）, upgrade（Cyan）...`。
**注意**：Python 中散落的布林（numbness/moving/anger/mouse_selected/been_targeted）
在 v2 統一收進 `statuses` 字典（見 §6），行為照 01/02 規格。

## 4. GameEvent（表現層協定）

至少支援（相容 01 §10）：

```
ATTACK  {from, to, delay}            # 撲擊/開火指示
HURT    {at, delay, post_health}
FLOAT   {at, amount, delay}          # 傷害/治療飄字
MOVE    {from, to}
DEATH   {at, delay}
SPAWN   {at, card_id, owner}         # v2 新增：入場演出
CAST    {at, kind}                   # v2 新增：施法演出（能力觸發時）
STATUS  {at, status_id, on|off}      # v2 新增：麻痺/怒氣圖示
RESOURCE{owner, kind, value}         # v2 新增：token/運氣/金幣/圖騰變化
```

表現層的 `CombatScheduler` 把事件按 `delay` 排到時間軸播放；
core 不等待動畫（AI 控制器靠「事件是否清空」判斷是否出手，同 Python renderer_busy 語義）。

## 5. 能力系統 v2（本次重構的核心目標）

### 5.1 觸發點（Trigger）——必須完整覆蓋 Python hook 集

| Trigger | 對應 Python hook | 時機 |
|---|---|---|
| ON_DEPLOY | `deploy()` | 棋子放置後 |
| ON_REFRESH | `on_refresh()` | 擁有者回合開始 |
| ON_SETTLE | `on_settle()` | 擁有者回合結束計分（可回傳分數修改，見 5.4） |
| ON_UPDATE | `update()` | 每 logic_step（動態 extra_damage 類） |
| ON_ABILITY_HIT | `ability(target)` | 傷害管線步驟 3「攻擊附帶」 |
| MOD_DAMAGE_BONUS | `damage_bonus()` | 管線步驟 4（回傳修改值） |
| MOD_DAMAGE_REDUCE | `damage_reduce()` | 管線步驟 5（回傳修改值） |
| MOD_FIELD_INTERCEPT | `on_field_effect_trigger()` | 管線步驟 6（全場、含 priority） |
| BLOCK_DAMAGE | `damage_block()` | 管線步驟 2（可整段取消） |
| ON_AFTER_DAMAGE | `after_damage_calculated()` | 管線步驟 9 |
| ON_BEEN_ATTACKED | `been_attacked()` | 管線步驟 8 |
| ON_KILLED | `killed(victim)` | 我殺了人 |
| ON_BEEN_KILLED | `been_killed(attacker)` | 我被殺 |
| ON_DIE | `die()` | 回收時 |
| CAN_BE_KILLED | `can_be_killed()` | 死亡判定（怒氣不死身） |
| ON_MOVE_BROADCAST | `move_broadcast(mover)` | 任何棋子移動後（全場廣播） |
| ON_AFTER_MOVEMENT | `after_movement()` | 自己移動後 |
| CUSTOM_MOVE | `custom_move()` | 攔截移動（保留） |
| ON_AFTER_ATTACK_BROADCAST | `after_attack_broadcast()` | 每次成功結算後全場廣播（目前無卡用，保留） |
| ON_TOKEN_GAINED / ON_TOKEN_DRAW | Blue `after_token()` / `token_draw()` | Token 引擎轉發 |
| ATTACK_OVERRIDE | `attack()` 覆寫 | 整個攻擊流程客製（ADCO/ADCC/Fuchsia 全家/APTG 禁攻…） |

### 5.2 資料結構

```gdscript
class_name Ability extends RefCounted
var id: String                  # "red_adc_grow"（沉默用）
var trigger: int                # Trigger enum
var tags: Array[int]            # TRIGGERED / PASSIVE / ON_ENTER / FACTION / MODIFIER
var priority: int = 0           # MOD_FIELD_INTERCEPT 排序用（APTF=20）
var condition: Callable         # func(ctx) -> bool
var effects: Array[AbilityEffect]

class_name AbilityEffect extends RefCounted
func execute(ctx: AbilityContext) -> Variant   # MOD_* 類回傳新數值
```

`AbilityContext`：`core, source(PieceState), target, value:int, extra:{}`。

`AbilityComponentV2`（掛在 PieceState 上，純資料）：
- `native: Array[Ability]`（由 `ability_registry.gd` 依 card_id 組裝）
- `granted: Array[Ability]`（**附魔** API：`grant(ability)` / `clear_granted()`）
- `silenced_ids / silenced_tags`（**沉默** API：`silence(id)` / `silence_tag(tag)` / `clear_silence()`）
- `dispatch(trigger, ctx)`：過濾沉默 → condition → 依 priority → 執行；
  MOD_* 類觸發器回傳鏈式修改後的 value。

沿用現有原型（`script/abilities/`）的概念與 API 命名，但整個搬進 core 重寫（原型掛在 Node 上，不合 D1）。

### 5.3 Faction 引擎

跨棋子的機制不塞在單卡能力裡，做成訂閱 core 事件的小引擎（每局實例化）：
- **TokenEngine**（Blue）：`gain(owner, n)` 逐顆結算 → 廣播 ON_TOKEN_GAINED → 滿 3 換抽 → 廣播 ON_TOKEN_DRAW。
- **LuckEngine**（Green）：`lucky_effects(target, flags)` 完整實作 02 的規則。
- **TotemEngine**（DarkGreen）：`engrave(owner, times)` 含 SPDKG 倍率。
- **CoinEngine**（Cyan）：上限 50、升級價格計算。
- **ShadowEngine**（Fuchsia）：鏡像生成/同步移動/佔格/死亡連動。
- **BrownEngine**：effects_disabled 集合管理與回合開始恢復。

### 5.4 計分掛勾

`on_settle` 在 Python 是多型方法（SPW 2 分、HFR 怒氣 0 分、方塊 0 分…）。
v2 做法：TurnEngine 結算時對每子先取 `base = has numbness ? 0 : 1`，
再走 ON_SETTLE trigger 讓能力修改（SPW +1、HFR/HFC 怒氣歸 0、CUBE 類恆 0），
最後照 01 §3 清 numbness/moving/怒氣。

### 5.5 沉默/附魔（未來功能的落地點）

- 沉默：`piece.abilities.silence_tag(Tag.TRIGGERED)` → 所有觸發式能力停擺；
  faction 引擎在廣播前查 `is_ability_active`。Brown 的 effects_disabled 直接改寫成內建沉默（tag=FACTION）。
- 附魔：`piece.abilities.grant(AbilityRegistry.make("enchant_fire"))`；
  granted 能力可帶 duration（status 到期時移除）。
- 這兩個 API 在 Phase 1 就要有測試覆蓋（用 Brown SPBR 當沉默的實例）。

## 6. Status（狀態）系統

| status_id | 對應 Python | 語義 |
|---|---|---|
| `numbness` | `numbness` | 01 §3 規則；UI 顯示鋸齒圖示（沿用現有 stun/sleep 圖） |
| `moving` | `moving` | 有移動權（等待選目的地） |
| `anger` | `anger` | 各卡自訂含義（HFR 不死身、ASSO 狂暴、HFO 顯示用…），UI 亮框 |
| `selected` | `mouse_selected` | 表現層互動狀態（不進存檔） |
| `targeted` | `been_targeted` | 表現層提示 |

Status 是 PieceState 上的字典 `{id: {value, duration}}`；能力/引擎透過 API 加減，
變動時發 STATUS 事件給表現層。**不要**沿用舊 Godot buff 節點樹。

## 7. 表現層與動畫插槽（目標 2）

### 7.1 棋子視圖 PieceView

單一場景 `piece_view.tscn`：
```
PieceView (Node2D)
 ├─ VisualRoot (Node2D)        # 佔位幾何 or 精靈，動畫只動這層
 │   ├─ PlaceholderShape       # Polygon2D + Label（職業形狀表見 02 附錄）
 │   └─ SpriteSlot (Sprite2D/AnimatedSprite2D)  # 之後放美術
 ├─ StatsOverlay               # HP/ATK/護盾/extra 標籤（沿用血條元件概念）
 ├─ StatusIcons                # numbness/anger/moving
 └─ AnimationPlayer / Tween 驅動
```

### 7.2 AnimationSet Resource（可擴充動作插槽）

```gdscript
class_name PieceAnimationSet extends Resource
@export var idle: AnimSpec          # 待機（循環）
@export var ready_attack: AnimSpec  # 準備攻擊（如拉弓）
@export var attack: AnimSpec        # 出手
@export var projectile: PackedScene # 投射物場景（可 null＝近戰撲擊）
@export var impact: PackedScene     # 命中特效（可 null）
@export var hurt: AnimSpec          # 被擊中
@export var death: AnimSpec         # 死亡
@export var cast: AnimSpec          # 施展能力
@export var extra: Dictionary       # 未來擴充（"victory"、"taunt"…）
```

- `AnimSpec`＝{animation 名 or Tween 程序名, duration, sfx 佔位}。
- **預設 fallback**（Phase 2 全棋子共用）：
  idle=呼吸縮放、ready=前傾 0.1、attack=向目標撲 0.32s（Python 節奏）、
  projectile=null 時直接撲擊；ADC 示範配置一個「箭矢」投射物（三角形 Polygon2D 飛行 + 命中閃光）證明插槽可用、
  hurt=白閃+抖動（沿用現有 hit_flash 概念）、death=淡出+縮小、cast=原地脈衝光圈。
- 之後美術到位：每張卡在 `data/animation_sets/<card_id>.tres` 指定自己的 AnimationSet，
  沒有的卡自動用 fallback——**換圖不改程式**。

### 7.3 CombatScheduler

收 `drain_events()`，按 delay 排程：ATTACK → 播 attacker 的 ready_attack+attack（有 projectile 就發射，
命中時間 = delay+0.32×0.55）；HURT → hurt 動畫+血條；FLOAT → 飄字；DEATH → death 動畫後釋放視圖；
MOVE → tween 位移。提供 `is_busy()` 給 AI 控制器。設定「動畫開關」時全部瞬時完成（Python 有此選項）。

### 7.4 HUD 需求清單（對齊 Python 戰鬥畫面資訊）

分數條與 win_threshold 標記、當前回合方、雙方計時器、攻擊/移動/治療/方塊次數、
手牌（含 Cyan `(+)` 切換鈕）、牌庫/棄牌數、運氣/token/圖騰/金幣顯示（依雙方牌組含該色才顯示）、
提示開關（card_hints）、動畫開關、AI 目標黃圈、攻擊範圍預覽（含 Fuchsia 鏡像範圍）、暫停/勝負畫面。

## 8. 測試策略

- `tests/run_tests.gd`：`SceneTree` script，掃描 `tests/test_*.gd` 執行 `run()`，
  統計 pass/fail，失敗 exit code 1。跑法：`godot --headless -s tests/run_tests.gd`（工作目錄＝專案根）。
- `tests/helpers.gd`：建 core、放子、給資源、快進回合等工具（翻譯自 Python `tests/helpers.py`）。
- 覆蓋優先序：01 §3 回合流程 → §6 傷害管線 → 每色卡牌（翻譯 `tests/test_card_*.py` 全部案例）→ AI。
- 展示層冒煙測試：開 battle 場景跑 3 回合腳本化輸入不噴錯即可（人工驗收為主）。
