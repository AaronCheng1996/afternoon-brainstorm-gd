## 1. 一張圖看懂資料流（Sim/View 分離，決策 D1）

```
玩家輸入（滑鼠/鍵盤）
   │
   ▼
場景腳本（battle.gd / draft.gd …「薄殼」）
   │  組出 GameAction（純資料）
   ▼
GameCore.dispatch(action)          ←── script/core：純邏輯，RefCounted，零 Node 依賴
   ├─ 改狀態：PieceState / PlayerState / BoardState / Statistics
   ├─ 跑規則：TurnEngine（回合）、Combat（傷害管線）、能力系統（ability/）
   └─ 累積 GameEvent（含動畫延遲 delay）到內部佇列
   │
   ▼
場景呼叫 core.drain_events() 取走事件
   │
   ▼
CombatScheduler 依 delay 排時間軸播動畫（PieceView / Projectile / 飄字）
   │  is_busy()==true 期間鎖輸入
   ▼
動畫播完 → 場景以 core 最終狀態「重建棋盤」重新同步 → 解鎖輸入
```

**鐵律**：`script/core` 下任何檔案不得 `extends Node`、呼叫 `get_tree()`、`load("res://scenes...")`。
`tests/test_integration.gd` 第 4 部分用靜態掃描守住這條線。
好處：headless 可測、AI 可整局模擬、回放/連線（Phase 11/12）直接搭在事件模型上。

## 10. 建議讀碼順序（review 路線）

1. **資料模型**：`game_config` → `piece_state` → `board_state`/`player_state` →（`statistics`、`rng_service`）
2. **核心流程**：`game_core`（先看 `dispatch` 分派表與 `_spawn_card`）→ `turn_engine`（回合與結算）→ `combat`（十步傷害管線，對照 01 §6）
3. **能力系統**：`trigger` → `ability`/`ability_effect`/`ability_context` → `ability_component`（沉默/附魔）→ `cards/white.gd`（最簡單）→ 挑一色深讀（推薦 `fuchsia.gd` 鏡像，最能考驗架構）
4. **事件→動畫**：`game_event` → `combat_scheduler` → `piece_view`（fallback 動畫）→ `projectile`/`impact_flash`
5. **場景薄殼**：`battle.gd`（輸入→dispatch→重建棋盤）→ `draft.gd` → `main_menu.gd`/`end_game.gd`
6. **測試**：`run_tests.gd` → `test_integration.gd`（整局範例最有讀碼價值）

## 11. Review 時可留意的慣例與陷阱

- **記憶體**：core 全是 RefCounted，Godot 無循環 GC——凡自我/循環參考處都用了 WeakRef
  （`ability_component.source`、shadow 的 `linker`）或事後 `hit_cards.clear()`（自傷情境，
  見 `dark_green.gd`/`brown.gd` 註解）。新增類似結構時要比照辦理，測試 runner 會報 orphan。
- **決定性**：所有隨機走 `RngService`；UI 的攻擊範圍預覽刻意「取同距候選集合、不呼叫
  detection」以免消耗 rng 污染對局（`battle.gd`）。
- **statuses/counters**：棋子布林狀態統一走 `statuses` 字典＋API，不要新增散裝 bool 欄位。
- **常數雙源**：`game_config.gd` 的值 Python 寫死在程式裡，改值要兩邊同步；
  其餘數值一律讀 `Balance`，不要硬編碼。
