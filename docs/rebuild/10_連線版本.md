## 1. 目標與範圍

**做什麼**：一台**專用伺服器**（使用者提供的主機，跑 headless Godot）承載大廳與多個房間；
玩家用戶端連上伺服器 → 開房/加房 → 選秀 BP → 對戰 → 終局，規則/平衡與本機版完全一致
（同一套 `script/core`），演出走既有 事件→`CombatScheduler`→`PieceView` 管線；
第三方可以**旁觀**任一允許觀戰的房間。

**資訊模型（D19，本規格的簡化樞紐）**：
- **公開**：盤面全部棋子與狀態、**雙方手牌明細**、雙方資源（運氣/藍球/圖騰/金幣）、分數/回合、
  棄牌堆、牌庫**張數**、統計。→ 雙方玩家與旁觀者看到**同一份**資訊。
- **隱藏**：**牌庫抽序與未來隨機**（＝RNG seed）。**（2026-07-18 D19 修訂）僅在對局進行中**存在伺服器、
  任何用戶端都拿不到（防實時模擬未來）；**對局結束後 seed 公開**——終局回放下載可含 seed（P12-18）。
  進行中的快照/事件仍永不含 seed（P12-4 不變）。單機/爬塔模式 seed 本即公開。
- 推論：**全房間共用單一公開快照，無需逐客端遮蔽**（v1 草案模型 B 的「per-client masking」
  整塊消失，工程大幅簡化）；本機版也依 D19 顯示雙方手牌（P12-2，先行、不依賴網路）。

**明確不做**：
- **不與 Python 版互通**（D17）：協定自訂（ENet＋RPC），不相容 Python 的 TCP 快照協定。
- **不做帳號系統**：暱稱自由輸入（進房時帶入），不做註冊/登入/好友。邀請＝把房碼傳給朋友。
- **不追求競技級防作弊**：D19 之後唯一可作弊面＝「改機看牌庫序」——已由 seed 不出伺服器
  在物理上封死；其餘（改機發非法 action）由伺服器權威 dispatch 全部擋下。夠正派。
- **不做中繼/打洞**：用戶端直連專用伺服器（LAN IP 或使用者開的公網入口），不需要 NAT 穿透。

## 2. 為什麼幾乎是現成的（可複用資產）

D1 的 Sim/View 分離讓連線＝「把單機資料流接一段網路」：

| 既有資產 | 對連線的意義 |
|---|---|
| `GameCore.dispatch(GameAction)` → `GameEvent` → `drain_events()`（D1） | 輸入/輸出都是純資料，可 JSON 化過網路；伺服器權威 dispatch＝把本機的呼叫搬到 server。 |
| `RngService`（一局一 seed） | seed 只存伺服器＝牌庫序物理隱藏（D19 唯一秘密）；server 端對局完全決定性。 |
| `ReplayLog`（P11-2） | 一局連線對戰＝server 端即時累積的 ReplayLog：終局自動存檔、回放免費；重連/旁觀中途加入用「快照＋續播」而非重放全流程。 |
| `battle.gd._rebuild_board()`／`_resync()` | 用戶端「收到權威狀態→重畫」的機制已在；P12-4 把資料源一般化（core 或快照 Dictionary）。 |
| `CombatScheduler.is_busy()` | 用戶端收事件照本機管線播動畫；is_busy 同時是輸入閘。 |
| `AIController.tick()` 模式（P10-4） | 「外部來源每幀餵 0/1 個 action」已驗證；遠端玩家＝另一個 action 來源。 |
| `DraftDispatcher`＋`auto_fill_and_advance`（P11-1） | BP 連線化＝把 dispatch 搬 server；逾時補牌本來就是純函式，server 執行後廣播即可。 |
| `CountdownTimer`（P11-1） | 回合/選秀計時搬 server 權威，用戶端只顯示剩餘秒。 |

**結論不變**：`script/core` 零改動（維持零 Node、規則不變、全套件綠）。新工程集中在
`script/net/`（訊息/傳輸/伺服器/房間）＋場景的「來源接線」＋大廳 UI。

## 3. 傳輸層（已定）

**Godot High-Level Multiplayer：`ENetMultiplayerPeer`＋自訂 `@rpc`（RELIABLE 有序）**。
- 伺服器：`create_server(port, max_clients)`；用戶端：`create_client(host, port)`。
- **不用**場景複製（`MultiplayerSpawner`/`MultiplayerSynchronizer`）——狀態在 GameCore
  （RefCounted），節點級複製不適用。
- 訊息一律走 `NetMessage`（型別常數＋Dictionary payload＋版本欄），JSON 序列化；
  大小可觀的只有快照（4×4 盤面＋兩手牌，仍極小）。
- **握手版本閘**：`hello{game_version, data_version, intent, nickname, token?}` →
  `welcome{...}` 或 `rejected{reason}`；`data_version`＝`Balance.data_version()`（J），
  不一致直接拒絕（規則/平衡不同步必炸，寧可擋在門口）。
- 心跳/延遲：週期 ping/pong 量 RTT，UI 顯示；逾時視為斷線（進 P12-10 重連流程）。

**為何不選其他**（v1 分析摘要）：裸 TCP 要自管分包/心跳且無 ENet 的低延遲可靠通道；
WebSocket/WebRTC 是瀏覽器/公網打洞才需要；Python 協定不沿用（D17）。

**MultiplayerPeer 抽象＝Steam 之路**：High-Level Multiplayer 的 peer 可替換
（`ENetMultiplayerPeer` → GodotSteam 的 `SteamMultiplayerPeer`），RPC 層不動——
peer 建立務必集中於單一工廠（見 §9.5 的設計紀律）。

## 4. 同步模型（已定：伺服器權威＋單一公開快照）

- **只有伺服器跑權威 GameCore**（每房一顆）。seed 由 server 產生、永不下發。
- 玩家操作 → 用戶端組 `GameAction` → 送 server → server `dispatch`（合法性＋回合閘＋定序）→
  - 成功：把 `drain_events()` 的 `GameEvent` 流**廣播全房**（玩家＋旁觀者同一份），
    各端照本機管線播動畫；
  - 失敗：只回傳原因給行動者（顯示訊息，同本機 `ActionResult`）。
- **公開快照**（P12-4）在關鍵點下發校正：入房/開局、重連、旁觀中途加入，以及
  **每次成功行動後**（D20，2026-07-19；原僅回合交接下發，實機發現手牌/資源要等回合交接才同步，
  裁定為缺陷）。快照＝D19 公開資訊全集，**不含 seed/牌庫序**、體積小，每動作一份頻寬可忽略。
  事件流負責演出與即時性，快照負責一致性兜底並讓手牌/資源/盤面**即時**同步——兩者資料源都是
  同一顆權威 core，不會分岔。客端於動畫播畢套用該校正快照（忙碌時暫存、播完即套）。
- 用戶端維持一份輕量鏡像（快照重建）供渲染/範圍預覽/己方合法性預檢；**最終裁決永遠在 server**。
- v1 草案的「模型 A（客端跑 core＋seed）」**廢棄**：seed 下發＝牌庫序外洩，違反 D19 的唯一隱藏項。

## 5. 伺服器程序與房間制（D18）

### 5.1 伺服器程序
- 同一份專案、headless 進入點：`script/net/server_main.gd`（SceneTree），
  啟動：`godot --headless --path <專案> -s script/net/server_main.gd -- port=5555 max_rooms=16`。
- 設定檔 `server_config.json`（埠、最大房間數、每房觀戰上限、席位保留秒數、回放存檔開關）；
  命令列參數可覆蓋。啟動腳本＋部署說明＝P12-11。
- server 不載入任何場景/圖形；只用 `script/core`＋`script/net`＋`Balance`。

### 5.2 房間（Room）
- 屬性：`room_id`（短房碼，邀請用）、名稱、**公開/上鎖**（上鎖＝需密碼；公開房也出現在列表、
  上鎖房出現在列表但帶 🔒，或可選「隱藏」只能憑房碼進）、觀戰開關＋上限、房主。
- 席位：P1/P2（房主預設 P1，可換位）；就緒狀態；旁觀者清單。
- 生命週期：`waiting`（等人/就緒）→ `drafting`（BP）→ `battling` → `ended`（終局統計）→
  重開（回 waiting，同成員）或解散。玩家全離開即解散。
- 大廳操作：列表（房名/狀態/人數/觀戰數/鎖標記）、重新整理、建房、憑房碼加入、加入列表房。
- **密碼**：進房驗證（play 與 spectate 都要）；錯誤回 `rejected{bad_password}`。
  傳輸為明文 JSON（區網/朋友圈定位，見 §9 部署注意）。

### 5.3 角色
- `player`（占席位）、`spectator`（唯讀）。房內廣播對兩者**同一份**（D19）；
  旁觀者的 action 一律被 server 拒絕（唯讀由 server 保證，不只 UI）。

## 6. 對局流程（BP→對戰→終局）

1. 兩席就緒 → server 進 `drafting`：`DraftAction` 經 server 權威 dispatch、狀態廣播；
   非編輯方與旁觀者看即時選牌（BP 本就公開）；選秀計時逾時由 **server** 跑
   `auto_fill_and_advance` 廣播結果。
2. BP 完成 → server 建 GameCore（seed 只在 server）→ 廣播開局快照 → `battling`。
3. 對戰：§4 流程；回合計時 server 權威（逾時 server 端 `end_turn` 廣播）。
4. 終局：server 廣播終局＋完整統計 export；存 ReplayLog（`replays/` server 端）；
   房間進 `ended`，可重開或解散。用戶端終局畫面同本機（統計/回放鈕沿用）。

首版即含 BP（P12-8），但實作順序仍 battle 骨幹先（P12-6）再 BP——房間在 BP 完成前
可用「雙方預設牌組、跳過 BP」的開發旗標先驗證對戰鏈路（P12-6 驗收用，正式流程不外露）。

## 7. 旁觀系統（首版，P12-9）

- 加入：大廳選房（或房碼）＋`intent=spectate`；上鎖房同樣驗密碼；受房間觀戰開關/上限控制。
- 資料：**與玩家同一份**公開快照＋事件流（D19：看得到雙方手牌，看不到牌庫序）。
  中途加入＝當前快照＋此後事件續播（不重放整局）。
- UI：唯讀 battle 場景（無行動列、手牌不可點、顯示「旁觀中」徽章＋觀戰人數）；
  BP 階段同樣可旁觀；終局可看統計。
- 退出/斷線：直接移除（不佔席位、不保留）。

## 8. 斷線重連（首版，P12-10）

- 入座時 server 發**席位 token**（隨機、單局有效），用戶端存於記憶體（非落盤）。
- 玩家斷線：席位保留 `seat_hold_seconds`（預設 60，可設）；房內顯示「等待重連」；
  計時器暫停（避免逾時懲罰掉線方）。
- 重連：`hello{token}` → 驗證 → 逐出殘留舊連線 → 收復席位 → 補送快照＋續播。
- 逾時未回：對手可選「判勝結束」或「再等」；房主可解散。旁觀者無此機制。

## 9. 部署（已定，2026-07-16 使用者提供）

- **主機**：Ubuntu、可常駐、**固定 IP、直連公網**（無需 NAT 穿透/中繼）。主機上有使用者
  其他專案在跑——埠號取破萬避免衝突。
- **埠**：預設 **24242/UDP**（可由 `server_config.json` 或命令列覆蓋；避開 Steam 用戶端的
  27000–27100 區段）。部署時 `ufw allow 24242/udp`。
- **用戶端連線**：伺服器是私人的、不做公開伺服器列表——用戶端內建**預設位址（寫死使用者
  主機 IP，settings 可手動改）**；朋友拿到遊戲即可直連。若日後 IP 有變動疑慮，可掛一個
  便宜網域/DDNS 名稱指過去（選配，非必要）。
- **執行**：Linux 版 Godot 4.7 `--headless` 跑本專案的 `server_main.gd`（或匯出 Linux
  dedicated-server preset）。**常駐＝systemd unit**（P12-11 附範本：自動重啟、開機自啟、
  journald 收 log）。
- **安全**：ENet 無內建加密，密碼/暱稱走明文 UDP——定位朋友圈同樂可接受；若日後公開見生人
  再評估 DTLS，列 Phase 12 之後選配。server 端輸入全部當不可信：`NetMessage` schema 驗證
  ＋dispatch 合法性雙層擋。伺服器與用戶端同 repo 同版本發佈，版本閘保證新舊不混連。
- 房間/觀戰上限：預設 max_rooms=16、每房觀戰 8（config 可調，使用者無特別要求）。

## 9.5 Steam 展望（使用者目標：未來上架 Steam）

**結論：現在的架構選型正好是對 Steam 最友善的一條路，但要守住一個設計紀律。**

- Godot 遊戲上 Steam 本身無障礙（匯出→SteamPipe 上傳；Steamworks 每款 App 100 美元入場費）。
  Steam 整合走社群標準 **GodotSteam**（GDExtension，支援 Win/Linux/Mac）。
- **關鍵**：GodotSteam 提供 `SteamMultiplayerPeer`——它是 Godot `MultiplayerPeer` 的實作，
  已併入 GodotSteam 4.x 主線。因為本規格用 **High-Level Multiplayer（MultiplayerAPI＋RPC）**，
  屆時把 `ENetMultiplayerPeer` 換成 `SteamMultiplayerPeer`，**所有 RPC 程式碼不用動**；
  還免費獲得 Steam 的 relay（SDR，玩家免開埠、隱藏 IP）、大廳/配對、好友邀請加入。
- **設計紀律（P12-3 落實）**：peer 的建立集中在單一工廠（如 `NetTransport.create_server/client`），
  全案只有這一處知道用的是 ENet——未來換 Steam 只改一個檔。
  位址概念抽象為「端點」（ENet=IP:port、Steam=SteamID），訊息層不碰。
- **已知限制**：GodotSteam 的 **dedicated server 建置目前尚未含 MultiplayerPeer**（官方標註
  開發中）。屆時兩條路：(a) Steam 版仍連我們的 ENet 專用伺服器（完全合規，常見做法）；
  (b) 改 Steam 大廳＋玩家主機（server_relay 經 host 中繼），私人伺服器退役。到時再選。

## 10. 對既有鐵則的自我檢查

- **`script/core` 零 Node 依賴（D1）不變**：網路層在 `script/net/`（可 extends Node 用 RPC）；
  core 只被 dispatch/drain_events/快照讀取。`test_integration.gd` 靜態掃描不受影響。
- **規則/平衡以 Python 為準不變**：連線不改規則；server 權威保證單一裁決源；全套件須綠。
- **D19 是資訊呈現決策（D16 範疇）**：不影響任何對局結果；已記入 06 行為疑義登記表備查。
- **決定性**：server 端 RngService seed 決定性＋ReplayLog 照舊；D4（不跨引擎一致）不受挑戰。
- **headless 可測**：net 層全部設計為 loopback（同程序 server＋client peer）可測；
  每個 P12 任務都有 headless 驗收（見 06）。

## 11. 客端接線（Phase 12 後段規劃，2026-07-18）

### 11.1 現況盤點

P12-2～P12-11 交付＋實機部署後（2026-07-18 使用者驗證：連線 ✓、建房 ✓、開始對戰 ✓）：

| 層 | 狀態 |
|---|---|
| 伺服器端（`script/net/`：RoomManager／NetGameServer／NetGameSession／NetDraftSession／catchup／token 重連／server 端 ReplayLog／Ubuntu 部署包） | **完成**，loopback 測試護體＋實機跑起來 |
| 大廳 UI（`scenes/online/online_lobby`：連線設定/大廳/建房/房內四面板） | **完成**（房內可按「開始」） |
| **連線對戰畫面**（battle 收快照/事件、送 action） | **未接**——「開始」後 server 對局在跑，玩家只看到房內狀態提示 |
| **連線選秀畫面**（draft 網路模式） | 未接（lobby「開始」暫走跳過 BP 的開發旗標） |
| **旁觀者唯讀畫面**／**重連 UX**／**終局→再戰閉環** | 伺服器端支撐已備（catchup／token／`force_waiting`），客端畫面未接 |

＝使用者體感「對戰內容還沒有連線版本」的正解。後段任務＝**06 P12-12～P12-17（P12-18 選做）**。

### 11.2 架構決策（後段全部任務共用）

1. **@rpc 節點路徑鐵則**（實機 bug f329077 的教訓）：`@rpc` 要求兩端 multiplayer 節點路徑一致。
   `NetClient` 常駐 `online_lobby` 之下**不得搬移／不得隨場景切換釋放**。
2. **子場景嵌入，不做 change_scene**：進對戰/選秀時把 battle/draft **instantiate 為 online_lobby
   的子節點**（大廳 UI 隱藏），結束後釋放子場景回房內面板。NetClient 與連線因此全程存活。
   （替代案「NetClient 提升 autoload」牽動兩端路徑與 server_main 啟動流程，收益低，不採；
   日後上 Steam 換 transport 時再一併評估。）
3. **battle 第四種模式 `net`**：既有 本機 hot-seat／vs AI／回放 之外新增。輸入端：`_do` 的
   net 分支**只 encode 送 server、絕不本地 dispatch**；輸出端：`battle_events` 進既有
   CombatScheduler、換手校正快照 resync（P12-4 已把盤面資料源一般化）。
4. **display 鏡像**：net 模式本地沒有權威 core。HUD/攻擊範圍預覽等讀取面以「快照 → display
   鏡像盤面」重建，重用既有讀取碼；**鏡像嚴禁被 dispatch**（唯一寫入者＝快照/事件）。
5. **輸入 gating 單點化**：「我的席位＝當前玩家 且 `!is_busy()` 且 非旁觀」才開放行動；
   旁觀＝永久唯讀＋徽章。被拒 action（`action_rejected`）顯示於既有訊息列。
6. **回合倒數顯示**：server 於換手/校正快照附 `remaining` 秒（相容性小追加），客端做顯示性
   本地倒數；權威判定（逾時 end_turn）恆在 server。
7. **流程閉環**：正式流程＝lobby「開始」→ `start_draft` → BP（net）→ 對戰（net）→ 終局統計
   （資料源＝終局快照統計 export）→ 回房（`ended`）→「再來一局」＝雙方再就緒 → server
   `force_waiting` 重開。P12-6 的跳過 BP 開發旗標於 P12-13 後降級為 debug、不再外露。
8. **重連 UX**：斷線 → 遮罩自動帶 token 退避重試 → welcome＋catchup → 依房態自動回到
   對戰/選秀子場景續玩；對手 held → 顯示「對方斷線等待重連（剩餘秒）」；逾時 → forfeit 判勝畫面。
9. **測試模式沿用**：headless 驗收用同程序 _Bus（場景級：兩個 battle/draft 場景＋瞬時模式
   打完整局，斷言與 server 快照一致、非法輸入零送信）；@rpc-over-ENet 的跨機驗證屬【人工】
   （P12-17 總驗收清單）。

## 附錄：Python 版連線功能盤點（僅供設計參考，協定不採用）

`core/network/`（`server.py` 439 行、`client.py`、`messages.py`）：
- **傳輸**：裸 TCP + 4-byte 長度前綴 + JSON（`_send_msg`/`_recv_msg`）；每客端一執行緒。
- **握手**：`hello`（intent=play/spectate、version、可選 token）→ `welcome`（role/state/scene/token）
  或 `rejected`（version_mismatch）。
- **角色**：`player1`/`player2`（play）、`spectator`/`god`（god_view 觀戰）；`host_seat` 可設。
- **重連**：`_peer_token`（16 byte urlsafe）；帶對的 token 回來 → 逐出舊連線、收復座位。
- **心跳**：`ping`/`pong` + RTT 量測、`heartbeat_timeout` 清理死連線、`net_info` 廣播延遲/觀戰數。
- **狀態散布**：`broadcast_state`（全體同一份）與 `broadcast_state_for`（逐客端遮蔽）、
  `broadcast_scene`（切場景）、`broadcast_game_over`、`log_transfer`（對局紀錄 base64 回傳）。
- Godot 版**取其設計精神**（版本閘、token 重連、心跳/延遲顯示、房間狀態機），
  **不取其協定/傳輸**；且因 D19（手牌公開）**不需要** `broadcast_state_for` 的逐客端遮蔽，
  只留單一 `broadcast_state`。
