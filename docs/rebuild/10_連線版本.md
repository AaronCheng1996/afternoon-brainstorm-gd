# 10 連線版本規格（Phase 12）

> 狀態：**草案，待使用者確認**（P12-1 產出）。本文件依 00 決策 **D17**（連線納入計畫、
> 主機權威、不與 Python 版互通）撰寫，把架構選型與待決問題攤開；經使用者拍板後，
> §7 的實作拆解才轉為 06 清單的 P12-2…P12-n。
>
> 規格出處縮寫沿用 06：R=01規則、D=04架構、J=05JSON。
> 參考來源：Python 版 `core/network/`（`server.py`/`client.py`/`messages.py`）——**僅作功能盤點**，
> 協定不相容、不互通（D17）。

---

## 1. 目標與範圍

**做什麼**：兩台 Godot 用戶端在**區域網路**內對戰一局「午後激盪」，規則/平衡與本機版完全一致
（同一套 `script/core`），過程演出走既有 事件→`CombatScheduler`→`PieceView` 管線。

**明確不做**（本輪界線，沿用 D8 判斷 + D17）：
- **不與 Python 版互通**：Python 端是自製 TCP + 全狀態快照協定；跨引擎對接不划算，且 RNG 不同源
  （D4），連抽牌序都不會一致。Godot 版自成一套協定。
- **不做網際網路配對/中繼**首版：先區網直連（手動 IP）。中繼（relay）/公網打洞列未來評估。
- **不追求防作弊到競技等級**：定位為「朋友區網同樂」。作弊阻力取決於 §4 選定的同步模型
  （模型 B 天生隱藏私有資訊，模型 A 於信任前提下接受洩漏）——此為**待使用者決策**的樞紐。

**本文件要回答的待決問題**（§8 有彙整）：傳輸層、同步模型（隱藏資訊處理）、大廳流程、
BP 是否連線化、重連/觀戰是否進首版。

## 2. 為什麼 Godot 版天生適合（既有可複用資產）

D1 的 Sim/View 分離讓連線幾乎是「把單機的資料流接一段網路」而已：

| 既有資產 | 對連線的意義 |
|---|---|
| `GameCore.dispatch(GameAction)` → 累積 `GameEvent` → `drain_events()`（D1） | core 的輸入是**純資料 GameAction**、輸出是**純資料 GameEvent**，兩者都可 JSON 化過網路。 |
| `RngService`（一局一 seed，所有隨機經它） | 同一 Godot 版本 + 同 seed + 同 action 序列 = **完全決定性**（D4 只否定「跨引擎」一致，同引擎同版本 OK）。這是「模型 A（動作鏡像）」的地基。 |
| `ReplayLog`（P11-2：seed + 雙方牌組 + action 流；`simulate()` 重建最終 core） | 連線的一局本質上就是一份**即時串流的 ReplayLog**。初次入場/重連/觀戰可用「從頭重播 action 流」直接重建到當前狀態，零額外序列化程式。 |
| `battle.gd._resync()`/`_rebuild_board()`（動畫播完後從 core 最終狀態重建棋盤） | 客端「收到權威狀態 → 重畫棋盤」的機制已存在，只需把資料源從本地 core 換成網路來的 core/快照。 |
| `CombatScheduler`（吃 `drain_events()`，`is_busy()` 鎖輸入） | 客端收到事件後照本機同一條管線播動畫，無需新演出程式；`is_busy()` 也是「本回合能否操作」的現成閘。 |
| `AIController.tick(core, now_ms, renderer_busy)`（P10-4，純邏輯輪詢） | 證明「外部驅動者每幀給 core 0/1 個 action」的模式可行；連線的遠端玩家等同「另一個 action 來源」。 |

**結論**：連線不需要動 `script/core`（維持零 Node、規則不變、全測試綠）。工程集中在
**一層新的網路轉接（`script/net/`）＋ battle/draft/menu 場景的多一種「來源」接線**。

## 3. 傳輸層選型

### 建議：Godot High-Level Multiplayer（`ENetMultiplayerPeer` + `MultiplayerAPI` 自訂 RPC）

- **主機**：`ENetMultiplayerPeer.create_server(port, max_clients)`，`get_tree().multiplayer.multiplayer_peer = peer`。
- **客端**：`create_client(ip, port)`。
- 訊息用**自訂 `@rpc` 方法**（`rpc_id(peer_id, ...)`）傳送，走**可靠有序**通道（`transfer_mode = RELIABLE`）
  ——對戰動作必須不丟不亂序。**不使用**場景複製（`MultiplayerSpawner`/`MultiplayerSynchronizer`）：
  我們的狀態是 GameCore（RefCounted，非場景樹節點），節點級複製不適用，也會綁死表現層。
- 連線事件用 `multiplayer.peer_connected/peer_disconnected/connected_to_server/connection_failed/server_disconnected` 信號。

**為何不選其他**：
- **裸 `ENetConnection`**：更低階，要自己管 peer/channel，沒好處。
- **裸 TCP（照抄 Python）**：要自己做長度前綴分包、心跳、執行緒——Godot 的 ENet 高階層免費給你這些，
  且 ENet 是 UDP-based 可靠通道，延遲特性優於 TCP。**唯一相容 Python 的路是裸 TCP，但 D17 已決定不互通**，
  所以沒有沿用 Python 協定的理由。
- **WebSocket/WebRTC**：瀏覽器/公網才需要，區網首版用不上。

**協定版本閘**：握手交換 `Balance.data_version()`（J，資料版本 hash）＋遊戲版本字串；不一致直接拒絕
（對齊 Python `version_mismatch` 的精神），避免兩端規則/平衡不同導致的不同步。

## 4. 同步模型（**本規格最關鍵的待決點**）

主機權威已定（D17：host 跑 GameCore）。分歧在「客端怎麼取得/維持狀態」，直接牽動**隱藏資訊
（手牌、牌庫抽序）與作弊阻力**。兩個候選：

### 模型 A — 決定性動作鏡像（client 也跑一顆 GameCore）

- 開局 host 把 `seed ＋ 雙方牌組` 發給 client；兩端各建 `GameCore.setup(...)`（同 seed）。
- 玩家操作 → 組 `GameAction` → **送 host**。host 用 `dispatch` 驗證合法性＋定序，合法則以**確認序號**
  廣播該 action。兩端收到確認 action 後套用到自己的 core（＝ `ReplayLog.simulate` 的每步邏輯），
  產生**相同事件**餵各自的 `CombatScheduler`，動畫一致。
- 入場/重連/觀戰＝把「開局 seed＋牌組＋至今的確認 action 流」重放（現成 `ReplayLog`）。
- **優點**：幾乎零新序列化程式；重播/重連天生具備；頻寬極省（只傳 action，約每筆數十位元組）。
- **致命缺點——隱藏資訊外洩**：client 端持有完整 core，seed 已知 → **可推算對手手牌與整副牌庫抽序**。
  改機的客端能直接看穿對手手牌與未來抽牌。對「朋友同樂」或許可接受，對稍認真的對戰是破口。

### 模型 B — 主機權威 + 逐客端遮蔽狀態（client 不跑權威 core，只算自己那側）

- 只有 host 跑權威 GameCore。client **不知道 seed**、**不持有對手手牌/牌庫**。
- host 每次 `dispatch` 後：把 `drain_events()` 的 `GameEvent`（公開資訊：出牌、攻擊、傷害、死亡、
  移動、資源變化…卡片在「被打出」時才揭露，本就公開）**廣播**給客端播動畫；並在關鍵點
  （入場、回合交接、重連、觀戰者加入）送一份**該客端視角的遮蔽快照**
  （自己的手牌明細 + 全盤公開棋子 + 雙方公開資源；**對手手牌只給張數**）。
- client 維護一份「夠用來渲染與驗證自己輸入」的輕量鏡像：全盤棋子（公開）＋自己的手牌/資源。
  攻擊範圍預覽只需公開盤面（本就不碰對手手牌），自己回合的合法著手只需自己手牌＋盤面——都拿得到。
- **優點**：隱藏資訊**在物理上**不到客端，改機也看不到；對齊 Python 的 `broadcast_state_for`
  逐客端遮蔽設計。
- **代價**：需寫 `GameCore → 遮蔽 Dictionary` 的序列化器與 client 端「從 dict 重建盤面」路徑
  （`_rebuild_board` 的資料源一般化）；比模型 A 多一塊工。

### 建議

**採模型 B 為目標架構**，但**用模型 A 的 `ReplayLog` 決定性做「host 端的錄影/重連基礎」**：
host 本來就有完整 core 與 action 流，可即時寫 `ReplayLog`（重連時 host 重建該客端狀態並補送遮蔽快照，
而非把 seed 交給客端）。這樣兼得「重連/觀戰便宜」與「隱藏資訊不外洩」。

> **這是要請使用者拍板的第一決策**：接受模型 A 的簡單（但 LAN 信任、手牌可被改機看穿），
> 或投入模型 B 的遮蔽（首版工程較大但正派）。建議 B；若使用者只想先快速驗證可玩性，
> 可先出**模型 A 的 battle-only 原型**再升級到 B（兩者 host 端邏輯共用，差別在「發 seed」還是「發遮蔽快照」）。

## 5. 大廳 / 房間流程（建議首版）

沿用 Python 的極簡直連精神，不做房間伺服器：

1. 主選單新增「**區網對戰**」入口 → 子面板：**建立主機** / **加入**。
2. **建立主機**：選座位（預設玩家執先手 P1，可切）→ 開 server（預設埠 5555，可改）→
   顯示本機區網 IP 與埠、等待對手。
3. **加入**：輸入主機 IP（＋埠）→ 握手（版本/資料版本閘）→ 進入等待/選秀/對戰。
4. 兩端就緒後：host 決定 seed、進入 **BP 或預設牌組**（見 §6）→ 對戰 → 終局統計（同本機流程）。

**探索（可選、次階段）**：UDP 廣播 beacon 讓「加入」頁自動列出區網內開著的主機（免手打 IP）。
首版可先只做手動 IP，beacon 列為 P12 後段小任務。

**大廳狀態同步**：座位、對手是否就緒、雙方 `data_version` 是否相符、目前延遲（ping）顯示。

## 6. 選秀（BP）連線化

BP 核心（`DraftState`/`DraftDispatcher`，本機版）同樣是「action 進、結果出」的純邏輯，可比照對戰接線：

- **DraftAction 同步**：可編輯方送 `DraftAction` → host 驗證（回合閘、12 上限、同名≤2、魔法≤3、
  階段前進）→ 廣播確認。BP 的資訊多為公開（雙方牌組是攤開選的），遮蔽需求比對戰小。
- **階段與「輪到誰」**：`DraftState` 三階段（p1 前 6 → p2 選 12 → p1 後 6）本就規定可編輯方；
  非編輯方 UI 鎖定、顯示「對方選牌中…」。
- **P11-1 選秀計時**已在：連線時逾時自動補牌（`DraftDispatcher.auto_fill_and_advance`）由 **host 權威執行**
  並廣播，避免兩端各自補出不同結果。

**待決**：BP 是否進**首版**？建議**分期**——先出 **battle-only（預設牌組或本機各自選好牌組帶入）**
最快摸到「線上打一局」，BP 連線化列**緊接的下一個子任務**。理由：battle 接線是連線的骨幹（動作/事件/
遮蔽/重連全在對戰驗證），BP 是同模式的複用，先把骨幹打穩風險最低。

## 7. 建議實作拆解（**待使用者確認後**才追加進 06）

以下為 P12-1 通過後的候選子任務。順序假設 §4 選 **模型 B**、§6 選 **battle-first**。
每個仍照 06 規範：獨立 commit、headless 驗收優先、UI 類「headless 過 + 視覺待人工過目」。

| 暫編 | 任務 | 驗收重點（headless 可測部分） |
|---|---|---|
| P12-2 | `script/net/` 轉接層骨架：`NetMessage`（型別＋JSON 化）、`NetHost`/`NetClient`（ENet 封裝、握手、版本/資料版本閘、信號轉發）。**不含 UI**。 | 兩個 peer 在同 headless 程序內連上、握手、版本不符被拒（用 loopback；`get_tree().multiplayer` 於 SceneTree harness 可測）。 |
| P12-3 | `GameCore → 遮蔽 Dictionary` 序列化器 + client 端「從 dict 重建盤面」（一般化 `_rebuild_board` 資料源）。 | 序列化→重建後，公開盤面/自己手牌一致；**對手手牌只剩張數**（遮蔽正確性斷言）。 |
| P12-4 | battle 接線（host 權威）：本地/遠端 action 都經 host `dispatch`；事件廣播→client `CombatScheduler`；回合交接送遮蔽快照。 | headless：兩個 in-process core 經網路轉接打完一局，**最終分數/回合/統計 export 完全一致**（比照 P11-2/P10-5 整合測試手法）。 |
| P12-5 | 大廳 UI：建立主機/加入面板、座位選擇、就緒/延遲顯示、版本不符提示。 | node tree 斷言 + 開關/填充；連線流程【人工】。 |
| P12-6 | 斷線重連（若首版納入，見 §8）：host 端 token 重發 + 重建該客端遮蔽狀態補送；client 重連退避。 | headless：client 掉線→帶 token 重連→狀態補齊一致。 |
| P12-7 | BP 連線化：DraftAction 同步 + 非編輯方遮蔽/鎖定 + 逾時 host 權威補牌廣播。 | headless：兩端跑完三階段，最終雙方牌組一致；逾時補牌只由 host 定。 |
| P12-8（選） | 觀戰 + UDP beacon 探索（見 §8 決策）。 | 觀戰者收遮蔽（無任一方手牌）快照能重建公開盤面；beacon 列表。 |

> 數量/順序依 §8 決策浮動；BP-first 或模型 A 會改動 P12-3/P12-4 的內容。

## 8. 待使用者決策清單（P12-1 的驗收＝這些拍板）

1. **同步模型（§4，樞紐）**：模型 A（簡單、LAN 信任、手牌可被改機看穿）vs 模型 B（遮蔽、正派、
   工程較大）。**建議 B**（可先出 A 原型驗可玩性再升級）。
2. **BP 是否進首版（§6）**：**建議 battle-first**，BP 緊接其後；或堅持首版即含 BP。
3. **斷線重連是否進首版**：Python 有完整 token 重連。模型 B 下重連＝host 重建遮蔽快照補送，成本中等。
   **建議納入**（區網對戰掉線常見，體感差很多）。
4. **觀戰是否進首版**：Python 有 spectator/god view。觀戰＝多一個唯讀客端（收「雙方手牌全遮」快照）。
   **建議延後**到 P12-8（首版先兩人對打穩定）。
5. **主機座位**：預設主機執 P1、客端 P2，且可切換（對齊 Python `host_seat`）。確認是否採此預設。
6. **傳輸層（§3）**：確認採 Godot High-Level Multiplayer + ENet + 自訂 RPC（不用場景複製）。
7. **探索方式（§5）**：首版手動 IP，UDP beacon 自動列表列為次階段小任務。確認可接受。

## 9. 對既有鐵則的影響（自我檢查）

- **`script/core` 零 Node 依賴（D1）不變**：網路層落在新的 `script/net/`（可 `extends Node` 用信號/RPC），
  core 只被 `dispatch`/`drain_events`。`test_integration.gd` 的靜態掃描不受影響。
- **規則/平衡以 Python 為準不變**：連線不改任何規則；host 權威保證只有一套 core 在裁決。全套件須維持綠。
- **決定性**：即使選模型 B，host 端仍靠 `RngService` seed 決定性（重連/錄影用 `ReplayLog`）；
  D4「不跨引擎一致」不受挑戰（本就同為 Godot）。
- **不與 Python 互通（D17）**：協定自訂、不相容 Python TCP，已在 §1/§3 明確。

## 附錄：Python 版連線功能盤點（僅供設計參考，協定不採用）

`core/network/`（`server.py` 439 行、`client.py`、`messages.py`）：
- **傳輸**：裸 TCP + 4-byte 長度前綴 + JSON（`_send_msg`/`_recv_msg`）；每客端一執行緒。
- **握手**：`hello`（intent=play/spectate、version、可選 token）→ `welcome`（role/state/scene/token）
  或 `rejected`（version_mismatch）。
- **角色**：`player1`/`player2`（play）、`spectator`/`god`（god_view 觀戰）；`host_seat` 可設。
- **重連**：`_peer_token`（16 byte urlsafe）；帶對的 token 回來 → 逐出舊連線、收復座位。
- **心跳**：`ping`/`pong` + RTT 量測、`heartbeat_timeout` 清理死連線、`net_info` 廣播延遲/觀戰數。
- **狀態散布**：`broadcast_state`（全體同一份）與 **`broadcast_state_for`（逐客端遮蔽）**、
  `broadcast_scene`（切場景）、`broadcast_game_over`、`log_transfer`（對局紀錄 base64 回傳）。
- Godot 版**取其設計精神**（版本閘、逐客端遮蔽、token 重連、心跳/延遲顯示），
  **不取其協定/傳輸**（改 ENet + RPC，且事件驅動而非全狀態快照）。
