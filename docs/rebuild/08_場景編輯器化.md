## 1. 去 v2 命名（P7-1 / P7-2）

- **目錄**：`script_v2/` → `script/`、`scenes_v2/` → `scenes/`（git mv 保留歷史）。
  全專案 `res://script_v2|scenes_v2` 路徑替換（.gd / .tscn / project.godot）。
- **類別**：所有 `class_name XxxV2` → `Xxx`（如 `GameEventV2→GameEvent`、`CombatV2→Combat`、
  `TriggerV2→Trigger`、`AbilityV2→Ability`、`PieceViewV2→PieceView`…共 29 個）。
  舊碼已刪，名稱無衝突。統一規則：識別字 `V2` 一律移除（`sed -E 's/([A-Za-z_])V2\b/\1/'`）。
- **註解**：`v2` 字眼移除或改寫（如「能力系統 v2」→「能力系統」；game_event.gd 的
  「避開舊碼 GameEvent 衝突」註解已無意義，改寫）。
- 歷史文件（進度日誌、06 已勾任務的備註）**保留原文**——它們是紀錄，不改寫歷史。
- 驗收：`--import` 無錯；全測試套件綠；launch 煙霧（main_menu 正常啟動）。

## 2. 場景編輯器化準則（P7-3 ~ P7-6 共同規範）

**目標**：打開任一 `.tscn`，編輯器裡就能看到完整 UI 骨架；美術可直接改節點、換圖，不讀程式。

1. **靜態節點進 `.tscn`**：背景、標題、Label、Button、Panel、容器（HBox/VBox/Grid）、
   圖層 Node2D（BoardLayer/FxLayer…）、棋盤格線——一律在場景檔宣告。
2. **動態集合仍由程式生成**，但必須生成到 `.tscn` 內宣告好的**容器節點**下：
   手牌鈕 → `HandBox`；展示館卡 → `ExhibitGrid`；棋子視圖 → `BoardLayer`；
   飄字/投射物 → `FxLayer`；圖表折線/長條 → `ChartLayer`/`BarsBox`。
3. **場景唯一名稱（Scene Unique Names）**：程式要取用的節點在編輯器打開
   `unique_name_in_owner`（`%` 前綴），腳本一律 `%NodeName` 取節點——
   美術搬動節點位置/層級不會弄壞程式。
4. 腳本用 `@onready var _x := %NodeName` 取代 `_ensure_built()` 程序建構；
   移除程序建構路徑（不留雙軌）。`configure()`/行動方法等對外 API 不變。
5. 佈局：HUD 控件掛 `CanvasLayer` 下、位置/字級/顏色由 `.tscn` 屬性定義（theme override 進場景檔）；
   Node2D 世界（棋盤、棋子）維持座標制。
6. **測試調整**：headless 測試由 `Script.new()` 改為 `load("....tscn").instantiate()`；
   節點取得走場景樹（`%` 名稱）。每個場景任務附「節點樹存在」斷言（instantiate 後檢查關鍵節點）。
7. 規則不動：場景腳本仍是薄殼（輸入→`GameCore.dispatch`、事件→動畫、狀態→UI），core 不受影響。

# §5 二輪：美術可編輯化（Phase 14 盤查表）

> **產出自 P14-1（2026-07-20）**，對應 `06_任務清單.md` Phase 14。
> 一輪（Phase 7，§2）解決的是「編輯器裡看得到骨架」；**二輪要解決的是「美術改得動數值」**——
> 骨架雖在 .tscn，但決定畫面長相的參數（棋盤幾何、顏色、動態項目樣式、特效節奏）仍寫死在
> `.gd` 常數裡，美術不讀程式就動不了。本章逐場景清點四類缺口並指派去處。
>
> **盤查基準**：`godot-polish` @ 89c3995，scenes/ 12 個 .tscn＋16 個 .gd、`script/view/` 5 個 .gd，
> 全套件 2709 passed／洩漏 0/0。本章為文件任務，零程式改動。

## §5.0 分類與去處（判準）

| 類 | 判準 | 去處 |
|---|---|---|
| (a) 位置/尺寸賦值 | 值**隨資料筆數/座標算出來**＝動態集合排版（合規，§2-2 允許）；值為**固定畫面座標或固定尺寸**＝靜態佈局 | 動態→保留，但把其中的**排版參數**（間距/欄寬/最小尺寸/邊距）改 `@export`；靜態→移進 `.tscn` |
| (b) `Color()` 字面量 | 依語意分四種：玩家色（P1/P2）、UI accent（警示/提示/選中）、特效暫態色、圖表色 | 玩家色與 accent→**theme 具名色**；特效/圖表→檔頭 `@export` 區；**派別色一律 `Balance.color_rgb`（已合規，不得硬編）** |
| (c) `.new()` 建構的視覺元素 | 元素**帶樣式**（字級/底色/最小尺寸/色染）＝應抽 item 模板場景；**純結構或純資料**（Node2D 圖層、Line2D 折線、邏輯物件）＝保留 | 前者→`*.tscn` item 場景＋`instantiate()`；後者→保留 |
| (d) 缺美術插槽 | 該視覺面目前**只有程序生成的幾何/色塊**，沒有「放圖即生效」的節點 | 補插槽節點（Sprite2D/TextureRect），預設隱藏，有圖才顯示 |

**共通鐵則**（承 06 Phase 14 節引言）：預設值＝現值（改完畫面與現況相同）；
`%` 場景唯一名稱與容器結構是程式契約，美術可搬位置/調樣式，**改名/刪節點屬程式任務**；
每任務全套件綠＋洩漏 0/0。

## §5.5 素材路徑慣例裁定（P14-1 決議，回填 P14-5）

06 P14-5 原文提議 `res://art/pieces/<card_id>.png`。**盤查發現本專案已有資產根 `res://img/`**
（71 個 png：`img/UI/buff`、`img/UI/attack_pattern`、`img/board`、`img/piece/*`），
且 `encyclopedia.gd` 的 `ATTACK_ICON_DIR` 已建立「查檔存在才用、否則跳過」的慣例，
`piece_view.tscn` 的 buff 圖亦掛在 `img/UI/buff/`。

**裁定：沿用 `res://img/`，不另開 `art/`**。P14-5 的路徑慣例改為：

- 棋子貼圖：`res://img/piece/card/<card_id>.png`
- 棋盤底圖：`res://img/board/skin_ortho.png`、`res://img/board/skin_iso.png`
- 場景背景：`res://img/UI/bg/<scene_name>.png`

理由：兩套資產根會讓美術不知道該丟哪；既有 71 個檔案已在 `img/`，遷移它們屬無收益的改動。
查檔一律用 `ResourceLoader.exists()`（同 encyclopedia:250），**`img/` 空或不存在時行為與現況完全相同**。
