## 鐵則（違反即算任務失敗）

1. **絕對不修改 Python 版的任何檔案**。它只是參考書。連格式化、修 typo 都不行。
   平衡數值的 canonical JSON 檔在 Python repo 的 `config/`，同步方式是「複製到 Godot repo」，不是反向。
2. **平衡數值與規則行為完全以 Python 版為準**（僅此二者；美術/UI 依 D16 已脫鉤）。
   發現 Python 行為與直覺不符時，照 Python 行為做，並記錄到 `06_任務清單.md` 的
   「行為疑義登記表」。唯一例外：`01_遊戲規則規格.md` §12 列出的「Python 已知 bug」，
   按該節的裁定行為實作。
3. **美術/UI 自由設計（D16，2026-07-11 修訂）**：畫面、佈局、操作、演出以 Godot 版
   易用性優先，不再模仿 Python 版。程式端以 Theme／向量／程序動畫／粒子打磨；
   **點陣美術素材不自行生成**，由使用者提供後經 SpriteSlot／PieceAnimationSet 插槽換入
   （機制見 `04` §7，換圖不改程式）。
4. 每個任務完成時必須通過該任務的**驗收標準**（多半是 headless 測試），不可跳過。
   UI/視覺類任務 headless 驗收通過即可勾選並註明「視覺待人工過目」，使用者回饋另開任務。
5. **表現層改動不得改變對局行為**：`script/core` 維持零 Node 依賴（D1，
   `test_integration.gd` 有靜態檢查守著）；任何 UI/美術任務完成時全測試套件必須綠。

## 重構目標（來自專案擁有者）

1. **可擴充架構**：技能（能力）不再寫死在棋子類別裡，改為獨立的能力系統，
   使「沉默（禁用能力）」「附魔（動態賦予能力）」等未來效果可以實作。
2. **強化演出**：每棋子有可擴充的動作插槽——待機、準備攻擊、攻擊（含投射物與命中特效，
   例如拉弓動作＋箭矢飛行）、被擊中、施展能力。先用幾何圖形佔位，之後直接換美術。
3. **功能對齊 Python 版**：對戰、選秀（BP）、CPU、統計圖表等。
   （2026-07-11 修訂：對齊範圍縮為「規則與平衡」；畫面與 UI 依 D16 自由設計。）
4. **雙版本平衡同步**：以 JSON 為單一數值來源。
5. **為爬塔模式鋪路**：Python 版已有完整 endless（爬塔）模式，Godot 架構要能承接它。

## 重大決策記錄

| # | 決策 | 理由 |
|---|---|---|
| D1 | **Sim/View 分離**：規則核心為純 GDScript（`RefCounted`，零 Node 依賴），對外只吐「遊戲事件」；場景層訂閱事件播動畫 | headless 測試、AI 需要整局模擬複製（beam search）、未來重播/連線都靠它 |
| D2 | **棋子資料驅動**：只有一個通用 Piece 場景，數值/能力由 JSON+Resource 組出來；不再是 90 個 .tscn/.gd | 平衡改 JSON 即生效；新增卡牌不用開場景 |
| D3 | **能力系統 v2**：以現有 `script/abilities/` 原型概念為基礎全面重寫，觸發點完整涵蓋 Python 的 hook 集（見 `04` §5） | 沉默＝停用能力、附魔＝動態掛能力 |
| D4 | **RNG 不追求跨引擎一致**：Python `random.Random` 與 Godot RNG 演算法不同，同 seed 不會同結果。規則層面對齊即可，不承諾洗牌序列一致 | 技術上不划算 |
| D5 | **佔位美術**：沿用 Python 的職業多邊形頂點表（`02` 附錄）＋顏色＋文字。動畫插槽先行 | 使用者要求 |
| D6 | **測試**：不引外部插件，用自製 headless 測試 runner（`godot --headless -s tests/run_tests.gd`），測試案例翻譯自 Python `tests/test_card_*.py` | 零依賴、可 CI |
| D7 | **平衡 JSON**：canonical 在 Python repo `config/`；Godot 用同步腳本複製到 `data/balance/` 並在載入時驗 schema、顯示資料版本 hash（見 `05`） | 使用者已決定用 JSON 溝通 |
| D8 | **LAN 連線本輪不做**：架構上（D1 事件模型）預留可能性，但不排任務 | 範圍控制；Python 端協定為自製 TCP，跨引擎互連是另一個工程 |
| D9 | **Claude（beam search）策略排在爬塔階段**，一般戰役 AI 不依賴它 | 它需要 GameCore 深拷貝；先讓一般 AI 上線 |
| D10 | **戰役模式（campaign）移除**（2026-07-11 使用者決策）：不再做獨立戰役；教學功能改由日後的爬塔模式承擔 | 範圍收斂；避免與爬塔重複投入 |
| D11 | **爬塔模式（endless）擱置**（2026-07-11）：複雜度高，本輪不做，保留為未來工作（將取代教學） | 先交付可玩的本機雙人核心 |
| D12 | **CPU AI 延後**（2026-07-11）：Phase 3 暫不執行，Phase 2 收官後直接做 Phase 6 收尾 | 使用者「CPU 可以再等一下」；先清理收編 |
| D13 | ~~**Phase 6 目錄改名延後**~~（2026-07-11）→ **被 D15 取代**：改名提前於 Phase 7 執行 | — |
| D14 | **場景編輯器化**（2026-07-11 使用者需求）：靜態 UI 骨架宣告進 `.tscn`（編輯器可視可編輯，美術可接手）；動態集合生成到宣告好的容器；程式以 `%` 場景唯一名稱取節點；測試改 instantiate 場景。規格見 `08_場景編輯器化.md` | 全程式生成使編輯器看不到東西，人工/美術無法接手 |
