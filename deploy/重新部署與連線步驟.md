## A. 再次部署步驟（程式碼有更新時）

用於：你在本機（Windows 開發機）改了程式碼、重新打包 zip 之後，要把新版本換上主機。

### 1. 本機打包

把整個專案資料夾（`AfternoonBrainstorming_godot/`）重新壓成 zip，上傳覆蓋主機的
`/tmp/AfternoonBrainstorming_godot.zip`（用 `scp`、`rsync`、或你原本的上傳方式都行）。

### 2. 主機上：停掉服務、換新程式碼、重新匯入

```bash
# 停掉正在跑的伺服器（避免解壓時檔案被鎖或狀態不一致）
sudo systemctl stop afternoon-brainstorm-server

# 清掉舊版、解壓新版（server_config.json 存在 user:// 目錄，不在專案目錄內，不會被清掉）
sudo rm -rf /opt/afternoon-brainstorm/*
sudo mkdir -p /tmp/abstage
sudo unzip -q /tmp/AfternoonBrainstorming_godot.zip -d /tmp/abstage -o
SRC=$(find /tmp/abstage -maxdepth 2 -name "project.godot" -printf "%h\n" | head -1)
sudo cp -a "$SRC"/. /opt/afternoon-brainstorm/
sudo chown -R abrainstorm:abrainstorm /opt/afternoon-brainstorm
sudo rm -rf /tmp/abstage

# 重新匯入（建 .godot/ 全域類別快取；也是最快的「有沒有 parse error」健檢）
sudo -u abrainstorm /opt/godot/Godot_v4.7-stable_linux.x86_64 \
  --headless --path /opt/afternoon-brainstorm --import
```

**看這一步的輸出**：如果出現 `SCRIPT ERROR` / `Parse Error` / `Compile Error` 字樣，**不要往下做**，
先把完整錯誤訊息貼給我——代表這次上傳的版本有問題（例如 zip 沒打包完整、或程式碼真的有錯）。
沒有紅字錯誤才繼續下一步。

### 3. 重啟服務、確認正常

```bash
sudo systemctl start afternoon-brainstorm-server
sudo systemctl status afternoon-brainstorm-server --no-pager
journalctl -u afternoon-brainstorm-server -n 20 --no-pager
```

**預期看到**（且行程持續 `active (running)`，不是啟動後又自己停掉）：

```
[server] 午後激盪 專用伺服器啟動：埠 24242（UDP）／最大房間 16／每房觀戰上限 8
[server] 遊戲版本 godot-0.1.0／資料版本 bal 4.3.0.0 @xxxxxxxxxxxx。Ctrl-C 或 SIGTERM 結束。
```

記下這行的**資料版本**（`bal 4.3.0.0 @hash`）——連線時用戶端這串必須完全一致，不一致會被
版本閘拒絕（見下方故障排解）。

### 4. 一行速查（下次直接複製整段跑）

```bash
sudo systemctl stop afternoon-brainstorm-server && \
sudo rm -rf /opt/afternoon-brainstorm/* && \
sudo mkdir -p /tmp/abstage && \
sudo unzip -q /tmp/AfternoonBrainstorming_godot.zip -d /tmp/abstage -o && \
SRC=$(find /tmp/abstage -maxdepth 2 -name "project.godot" -printf "%h\n" | head -1) && \
sudo cp -a "$SRC"/. /opt/afternoon-brainstorm/ && \
sudo chown -R abrainstorm:abrainstorm /opt/afternoon-brainstorm && \
sudo rm -rf /tmp/abstage && \
sudo -u abrainstorm /opt/godot/Godot_v4.7-stable_linux.x86_64 --headless --path /opt/afternoon-brainstorm --import && \
sudo systemctl start afternoon-brainstorm-server && \
sleep 2 && journalctl -u afternoon-brainstorm-server -n 10 --no-pager
```

## B. 目前的連線步驟（用戶端怎麼連上）

### 1. 打開遊戲（Godot 編輯器跑專案，或未來匯出的執行檔）

- 入口場景：`scenes/menu/main_menu.tscn`
- 主選單 →「線上對戰」

### 2. 連線設定面板

填三欄：
- **暱稱**：隨意
- **伺服器位址**：你的主機公網 IP（不是 `127.0.0.1`，那是內建預設值，只在本機測試才用得到）
- **埠**：`24242`

按連線。（這幾欄會存到 `user://settings.json`，下次開遊戲會記住。）

### 3. 預期結果

- **成功**：畫面切到大廳，能看到房間列表（一開始是空的）、可以「建房」或輸入房碼加入。
- **連線失敗**（畫面顯示錯誤訊息，常見原因）：
  | 訊息/現象 | 原因 | 處理 |
  |---|---|---|
  | `無法建立連線` | 位址/埠打錯，或伺服器根本沒在跑 | 先在主機跑 `sudo systemctl status afternoon-brainstorm-server` 確認 `active (running)` |
  | 卡在「連線中…」很久後無反應 | 埠被擋（防火牆/雲端安全群組），或伺服器/用戶端版本裡的 RPC 節點路徑又對不上 | 先用下方「C. 只驗證伺服器活著（不透過遊戲）」排除是不是伺服器端問題 |
  | 明確被拒絕、顯示版本不符 | 用戶端與伺服器不是同一份程式碼（`game_version`/`data_version` 對不上） | 確認雙方都是「A. 再次部署」跑過的**同一次**打包內容 |

### 4. 目前能玩到哪裡（誠實告知現況）

**能驗證**：連線、大廳列表、建房/加房、雙方就緒狀態、延遲（RTT）顯示。

**還不能玩到**：正式對戰畫面（出牌/攻擊/移動）目前**還沒接線到連線版**——房內按「開始」
只會顯示狀態文字，看不到棋盤。這塊（`draft.tscn`／`battle.tscn` 收快照/事件並播動畫）是
下一階段的工作，不是這次部署的範圍。如果你想推進到「真的能連線對戰」，跟我說一聲，我可以
接著做這塊接線。

## C. 只驗證伺服器活著（不透過遊戲，純檢查用）

不想開遊戲、只想快速確認伺服器狀態正常時：

```bash
# 服務狀態
sudo systemctl status afternoon-brainstorm-server --no-pager

# 最近的 log
journalctl -u afternoon-brainstorm-server -n 30 --no-pager

# 埠有沒有真的在聽（UDP）
sudo ss -ulnp | grep 24242
```

`ss` 那行如果有輸出且行程名是 Godot，代表埠確實開著、由伺服器行程持有。

## 已知的部署陷阱（踩過的坑，之後遇到類似狀況先查這裡）

1. **`anim_demo.gd` parse error**（2026-07-18 發現並修復）：`--headless --import` 報
   `Cannot infer the type of "tw" variable`。已修正，若還看到這個錯誤代表上傳的版本太舊。
2. **連線卡住、log 出現 `Node not found: "OnlineLobby/NetPeer"`**（2026-07-18 發現並修復）：
   用戶端與伺服器的 `@rpc` 節點路徑對不上。已修正，若還看到這個錯誤同樣代表版本太舊。
3. **`server_main.gd` 直接跑會報 `Identifier not found: Balance`**（P12-11 已修復）：
   同上，太舊的版本才會這樣。
4. **房內/對戰顯示「玩家NNNN」短碼而不是對方暱稱**（P12-21，2026-07-19）：
   對手暱稱是由**伺服器**在房態廣播裡附帶 `names` 欄位下發的（P12-17 才加）。若主機上的
   server 早於 P12-17，就不會外送 `names`，用戶端只好退回短碼。**解法＝照「A. 再次部署步驟」
   重佈最新版**。（P12-21 後客端已保證不再出現裸 peer id 如 `對手 #948868441`；但要看到
   *真正的暱稱*，伺服器必須是 P12-17 以後的版本。）另：暱稱欄留空時客端會自動填
   「玩家NNNN」預設，故空暱稱也不會再變成裸編號。

以上都已在目前版本修好；只要照「A. 再次部署步驟」用最新打包的 zip 上去，理論上不會再遇到。
